# START HERE

Quick orientation for anyone working on `xmtp_plugin`.

## What This Is

A Flutter plugin providing XMTP decentralized messaging across Android, iOS, macOS, Windows, and Web. Published on pub.dev as `xmtp_plugin`. The Dart API (`XmtpPlugin` class) is the same on every platform; each platform has its own native backend.

## Architecture

```
Dart (XmtpPlugin)
  |
  v
XmtpPluginPlatform (abstract interface)
  |
  +-- MethodChannelXmtpPlugin -----> Android (Kotlin, xmtp-android SDK)
  |                                  iOS (Swift, XMTP iOS SDK)
  |
  +-- XmtpPluginWindows -----------> Rust FFI (flutter_rust_bridge -> libxmtp)
  |
  +-- XmtpPluginWeb ---------------> JavaScript interop (XMTP Browser SDK v5)
```

The codec system (encode/decode messages) lives in the Dart layer (`codecs.dart`), not in platform code. Platform implementations only transport raw bytes.

## Key Files

| File | What it does |
|------|-------------|
| `lib/xmtp_plugin.dart` | Public API. All methods delegate to platform interface |
| `lib/xmtp_plugin_platform_interface.dart` | Abstract contract every platform implements |
| `lib/xmtp_plugin_method_channel.dart` | Android/iOS implementation via MethodChannel |
| `lib/xmtp_plugin_windows.dart` | Windows implementation, calls Rust via dart:ffi |
| `lib/xmtp_plugin_web.dart` | Web implementation, calls JS via dart:js_interop |
| `lib/codecs.dart` | 13 codec implementations + registry + content type models |
| `lib/src/rust/` | Auto-generated by flutter_rust_bridge. Do not edit manually |
| `lib/generated/` | Protobuf-generated Dart. Regenerate from `proto/` if changed |
| `rust/src/api/` | Rust API modules exposed to Dart via FRB |
| `rust/src/api/client.rs` | Client init, key generation, environment resolution |
| `rust/src/api/messaging.rs` | Send/receive messages, streaming |
| `rust/src/api/conversations.rs` | List/create/sync conversations |
| `rust/src/api/groups.rs` | Group CRUD, member/admin management, inbox states |
| `rust/src/api/consent.rs` | Consent per conversation and per inbox |
| `rust/src/api/inbox.rs` | Installation ID, inbox state, sync request |
| `rust/src/api/signing.rs` | EIP-191 signing, revoke installations, add/remove accounts |
| `rust/src/api/helpers.rs` | Client accessor, topic mapping, time conversion utilities |
| `android/src/main/kotlin/io/xmtp/xmtp_plugin/XmtpPlugin.kt` | Kotlin method channel handler |
| `ios/Classes/XmtpPlugin.swift` | Swift method channel handler |
| `web/xmtp_client_manager.js` | JavaScript facade wrapping XMTP Browser SDK v5 |

## Windows DLL Deployment (Critical)

The pub.dev package does **not** include a pre-built DLL. Consumers of the package on Windows must clone this repo, build the Rust crate, and copy the DLL next to their app executable. See the README for the full step-by-step.

- Debug DLL: `rust/target/debug/xmtp_plugin_native.dll` → `<app>/build/windows/x64/runner/Debug/`
- Release DLL: `rust/target/release/xmtp_plugin_native.dll` → `<app>/build/windows/x64/runner/Release/`

If the DLL is missing, the app crashes immediately with no clear error. If it's <1MB it's a stale CMake stub — rebuild with `cargo build`.

## Gotchas

### flutter_rust_bridge version mismatch
The Dart and Rust FRB versions must be identical. Cargo.toml pins `flutter_rust_bridge = "=2.11.1"`. The pubspec.yaml must pin `flutter_rust_bridge: 2.11.1` (no caret). If the Dart side resolves to 2.12.0, the app crashes at launch with: "codegen version (2.11.1) should be the same as runtime version (2.12.0)".

### Kill app before rebuilding
The Windows linker cannot overwrite the exe while the process is running. You get `LNK1168: cannot open ... for writing`. Kill the app first.

### First Rust build is slow
Compiling libxmtp from scratch takes 6+ minutes (500+ crates). Incremental builds after that are fast (~10s for API changes). The build requires Strawberry Perl and ATL headers in Visual Studio.

### DLL size check
A valid debug DLL is ~58MB. If it's <1MB it's a stale stub from CMake, not a real build. Rebuild with `cargo build` from the `rust/` directory.

### Singleton client
Only one XMTP client is active at a time. Calling `initializeClient` replaces the previous client. The Rust side stores the client in a global `LazyLock<Mutex<Option<ClientState>>>`.

### History sync is unreliable
`sendSyncRequest()` triggers history transfer between installations but it doesn't complete reliably on the current XMTP network. Do not depend on it for tests or critical flows.

### Environment parameter
`initializeClient` accepts an `environment` parameter: `'production'` (default), `'dev'`, or `'local'`. This is threaded through all platforms. The Rust code stores the environment in `ClientState` so subsequent API calls (inbox lookup, static methods) use the same network.

### FRB generated code
Files in `lib/src/rust/` and `rust/src/frb_generated.rs` are auto-generated by flutter_rust_bridge. After changing function signatures in `rust/src/api/*.rs`, run `flutter_rust_bridge_codegen generate` to regenerate. If you manually edit generated files (e.g. to add a parameter), the function ID (`funcId`) and serialization order must match exactly between Dart and Rust.

### Android package namespace
The Android plugin uses `io.xmtp.xmtp_plugin`. This is reflected in `android/build.gradle`, `AndroidManifest.xml`, and the Kotlin source directory structure.

### Method channel name
All platforms use the method channel name `xmtp_plugin`. This string must match in Dart (`MethodChannel('xmtp_plugin')`), Kotlin, and Swift.

## Adding a New API Method

1. Add the Rust function in `rust/src/api/<module>.rs`
2. Run `flutter_rust_bridge_codegen generate` to update `lib/src/rust/`
3. Add the method to `XmtpPluginPlatform` (abstract interface)
4. Implement in `MethodChannelXmtpPlugin` (for Android/iOS via method channel)
5. Implement in `XmtpPluginWindows` (calls the generated Rust binding)
6. Implement in `XmtpPluginWeb` (calls the JS facade)
7. Add the Kotlin handler in `XmtpPlugin.kt`
8. Add the Swift handler in `XmtpPlugin.swift` (iOS)
9. Add the JS method in `web/xmtp_client_manager.js`
10. Expose in `XmtpPlugin` (the public Dart class in `xmtp_plugin.dart`)

## Adding a New Codec

1. Create a class extending `XMTPCodec` in `codecs.dart`
2. Implement `encode()` and `decode()` with the `EncodedContent` protobuf
3. Register it in `initializeClient()` or let users register it manually
4. If it needs a protobuf definition, add a `.proto` in `proto/content_types/` and regenerate

## Dependencies

### Dart
- `plugin_platform_interface` for the platform abstraction pattern
- `flutter_rust_bridge: 2.11.1` for Windows FFI (pinned, must match Rust)
- `protobuf` for message content encoding
- `web3dart`, `crypto`, `pointycastle` for Ethereum crypto in codecs
- `http` for remote attachment downloads
- `web` for web platform support

### Rust (Cargo.toml)
- `xmtp_mls`, `xmtp_id`, `xmtp_db`, `xmtp_api_grpc`, etc. from libxmtp v1.9.0
- `flutter_rust_bridge = "=2.11.1"`
- `k256`, `sha3`, `rand` for secp256k1 crypto
- `tokio` for async runtime
- `reqwest` for HTTP (remote attachments)
- Patched: `diesel` (ephemera fork), `hpke-rs` (pinned v0.4.0 in `rust/deps/`)

### Android (build.gradle)
- `org.xmtp:android:4.7.0`
- `org.web3j:core:4.9.7`
- `com.google.protobuf:protobuf-javalite:3.21.12`

### iOS (podspec)
- `XMTP ~> 4.0`

### Web (package.json)
- `@xmtp/browser-sdk ^5.0.1`

## Test App

`test_app/` is a Windows Flutter app that runs 21 integration test steps against the XMTP dev network. It auto-runs on launch and writes results to `%TEMP%/xmtp_test_results.log`.

Tests cover: key generation, client initialization on dev network, DM creation and messaging, group creation with metadata, message verification across accounts, addAccount identity linking, and inbox ID verification across linked keys.

To run:
```powershell
cd test_app
flutter pub get
flutter run -d windows
# Check results:
type %TEMP%\xmtp_test_results.log
```
